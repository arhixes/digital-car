# Бортовой Ассистент Автомобилиста (Digital Car Manual)

## 1. Проблема

Владелец автомобиля сталкивается с рядом типовых сложностей при эксплуатации и обслуживании машины:

- Информация по обслуживанию, ремонту и подбору запчастей разбросана по разным источникам (официальные сервис-мануалы, таблицы допусков, форумы), и найти нужный ответ быстро — трудозатратно.
- Владелец часто забывает о сроках ТО, пока не столкнётся с поломкой или просроченной гарантией.
- Решения по сезонным вопросам (замена шин на зимние/летние, стоит ли ехать на мойку) принимаются «на глаз», без учёта фактических погодных данных.
- Обычные LLM-чат-боты общего назначения дают неточные или «усреднённые» ответы, не привязанные к конкретной модели авто и её реальному регламенту, что создаёт риск ошибок при ремонте/обслуживании.

Нужен инструмент, который совмещает точные знания из официальной документации конкретного автомобиля с проактивными вспомогательными функциями и удобным каналом уведомлений.

## 2. Цель

Разработать пайплайн загрузки данных в RAG-хранилище и AI-агента, который:

1. Отвечает на вопросы пользователя по эксплуатации и ремонту конкретной модели автомобиля, опираясь на загруженные мануалы и справочные материалы.
2. Использует набор инструментов для проактивных задач: расчёт пробега до ТО, отслеживание погодных условий для рекомендаций по шинам и мойке.
3. Работает по расписанию для периодической проверки условий и отправляет уведомления пользователю через Telegram-бота.

## 3. Функциональные требования

### 3.1. Вход

| # | Источник | Формат | Описание |
|---|----------|--------|----------|
| 1 | Сервисный мануал выбранной марки/модели авто | PDF | Полное руководство по ремонту и эксплуатации |
| 2 | Таблицы подбора запчастей | PDF / XLSX | Допуски масел, размеры дисков, моменты затяжки и т.п. |
| 3 | Дополнительные материалы | PDF / HTML / текст | Официальные бюллетени, статьи, обсуждения на форумах |
| 4 | Пользовательский запрос | Текст | Вопрос по эксплуатации/ремонту |
| 5 | Данные пользователя о ТО | Число (пробег, км) / дата | Текущий пробег, дата последнего ТО |
| 6 | Геолокация пользователя (для погоды) | Город | Нужна для запроса прогноза погоды |
| 7 | Настраиваемые пороговые параметры | JSON | Например: температура для смены шин, порог осадков для мойки |

### 3.2. Выход

| # | Тип | Формат | Описание |
|---|-----|--------|----------|
| 1 | Ответ агента на вопрос | Текст (Telegram-сообщение) | Ответ, сгенерированный на основе RAG-контекста, со ссылкой/цитатой на источник (раздел мануала) |
| 2 | Напоминание о ТО | Telegram-сообщение | «До следующего ТО осталось X км / Y дней» |
| 3 | Рекомендация по шинам | Telegram-сообщение | «Среднесуточная температура ниже +5°C — рекомендуется замена на зимние шины» |
| 4 | Рекомендация по мойке | Telegram-сообщение | «Ожидается дождь — мойку лучше отложить» / «Погода благоприятна для мойки» |
| 5 | Лог/история взаимодействий | БД (для отладки/аналитики) | История вопросов-ответов и сработавших триггеров |

## 4. Пайплайн

### 4.1. Пайплайн подготовки RAG (offline / по запросу пользователя)

```
[1] Выбор марки/модели авто
        ↓
[2] Ручной сбор источников (Service Manual, таблицы запчастей, форумы)
        ↓
[3] Загрузка файлов в систему
        ↓
[4] Парсинг и извлечение текста
    - PDF → текст (+ таблицы отдельно)
    - HTML → очистка от разметки
        ↓
[5] Нормализация и очистка текста (удаление мусора, дублей)
        ↓
[6] Чанкинг (разбиение на смысловые фрагменты, с учётом заголовков разделов
    и сохранением таблиц как отдельных чанков/структурированных записей)
        ↓
[7] Обогащение метаданными (модель авто, раздел мануала, тип источника,
    дата/версия документа, номер страницы)
        ↓
[8] Векторизация
        ↓
[9] Загрузка в векторное хранилище (Vector DB / Qdrant), индексация
        ↓
[10] Создание и настройка Telegram бота через aiogram
        ↓
[11] Установка бота на хостинг

```

### 4.2. Пайплайн ответа на вопрос (online, RAG-агент)

```
Пользователь → сообщение в Telegram
        ↓
Telegram Bot → передаёт запрос агенту
        ↓
Агент
   - анализирует вопрос по мануалу / запрос статуса ТО / запрос погоды / прочее
        ↓
   ├── Если это вопрос по эксплуатации/ремонту:
   │     → Поиск релевантных чанков в Vector DB / Qdrant (по модели авто + запросу)
   │     → Формирование промпта с контекстом
   │     → Генерация ответа LLM со ссылкой на источник/раздел
   │
   ├── Если запрос требует инструмента:
   │     → Вызов инструментов
   │     → Получение результата инструмента
   │     → Формирование ответа на основе результата
        ↓
Ответ агента → Telegram Bot → пользователь
```

## 5. Алгоритм работы

1. **Инициализация профиля автомобиля**: пользователь выбирает марку и модель авто.
2. **Сбор источников**: вручную находятся и скачиваются сервисные мануалы, таблицы допусков, дополнительные материалы.
3. **Загрузка в RAG**: файлы прогоняются через пайплайн подготовки (парсинг → чанкинг → эмбеддинги → индексация в БД), привязываясь к конкретной модели авто.
4. **Настройка агента**: агенту передаются доступные инструменты (`Maintenance_Scheduler`, `Weather_Control`) и системный промпт с инструкцией отвечать строго на основе найденного в RAG контекста, а не общих знаний.
5. **Настройка Telegram-бота**: связывается с агентом как единая точка входа/выхода для пользователя.
6. **Настройка расписания**: задаются интервалы проверки для фоновых задач ТО и погоды, а также пороговые параметры (температура, осадки).
7. **Основной цикл работы**:
   - Пользователь пишет вопрос в Telegram → агент классифицирует запрос → либо делает поиск и генерирует ответ, либо вызывает нужный инструмент → отправляет ответ пользователю.
   - Параллельно, по расписанию, фоновые задачи проверяют условия (пробег, погода) и, если срабатывает условие, инициируют уведомление через того же Telegram-бота.
8. **Логирование**: каждый запрос/ответ и срабатывание тулов сохраняются для последующего анализа качества и отладки RAG (какие вопросы не находят хорошего контекста и т.д.).

## 7. Реализация интерфейса (Telegram-бот: кнопки и FSM)

Не все функции требуют LLM. Свободные текстовые вопросы по мануалу идут через RAG-агента, а рутинные операции (ввод пробега, настройка порогов, проверка статуса ТО и погоды) реализуются как обычная кнопочная логика — без обращения к нейросети.

### 7.1. Два независимых контура

- **Rule-based контур** (кнопки, FSM, простая логика сравнения/арифметики) — разделы «ТО» и «Погода/шины», настройки. LLM не используется.
- **RAG-контур** — единственная точка входа: кнопка «Вопрос по авто». Здесь пользователь пишет свободный текст, который уходит в агента (retrieval + генерация ответа).

### 7.2. Структура меню

**Главное меню (Reply-кнопки, всегда на экране):**

```
[ ТО ] [ Погода/шины ]
[ Вопрос по авто ]
```

**Раздел «ТО» (Inline-кнопки):**

```
[ Указать текущий пробег ]
[ Указать дату последнего ТО ]
[ Проверить статус ТО ]
[ ⬅ Назад ]
```

**Раздел «Погода/шины» (Inline-кнопки):**

```
[ Установить порог температуры (шины) ]
[ Проверить: пора менять шины? ]
[ Ехать на мойку сегодня? ]
[ ⬅ Назад ]
```


### 7.3. Пример FSM-сценария «Указать пробег»

```
Пользователь → кнопка «Указать текущий пробег»
        ↓
Бот переводит в состояние waiting_for_mileage
        ↓
Бот: «Введите текущий пробег в км:»
        ↓
Пользователь: «87000»
        ↓
Валидация (is digit, разумный диапазон) — без LLM
        ↓
Сохранение в БД
        ↓
Бот: «Пробег сохранён»
```

### 7.4. Логика инструмента `Maintenance_Scheduler`

- Интервал ТО для конкретной модели (например, «каждые 10 000 км или 12 мес») извлекается из мануала один раз на этапе загрузки данных (вручную или через RAG) и сохраняется в конфиг/таблицу по модели — не через RAG-поиск при каждой проверке.
- Далее сравнение с текущим пробегом, формирование сообщения по шаблону.

### 7.5. Логика инструмента`Weather_Control`

- Запрос в погодный API (например, OpenWeatherMap) по городу пользователя.
- `if avg_temp < threshold:` → рекомендация по шинам (шаблонное сообщение).
- `if probability_of_rain > X%:` → рекомендация по мойке (шаблонное сообщение).
- Никакой генерации текста не требуется — используются заранее заготовленные шаблоны.

### 7.6. Итоговая схема веток

```
/start
  │
  ├─ ТО ──────────────► FSM (ввод пробега/даты) ──► БД ──► арифметика ──► ответ шаблоном
  │
  ├─ Погода/шины ─────► FSM (настройка порога) ──► Weather API ──► if/else ──► ответ шаблоном
  │
  ├─ Вопрос по авто ──► свободный текст ──► RAG ──► LLM ──► ответ
  │
  └─ Настройки ───────► FSM (город, модель авто и т.п.) ──► БД
```

## 8. MVP (Minimum Viable Product)

**Цель MVP:** доказать, что базовый RAG-механизм работает — агент может отвечать на вопросы пользователя по одной конкретной модели авто, опираясь на загруженный мануал.

### Что входит в MVP

- Telegram-бот
- Один автомобиль: **Hyundai Elantra 2004**
- Инструменты  `Weather_Control` и `Maintenance_Scheduler`
- Пайплайн загрузки: PDF → текст → чанкинг → эмбеддинги → векторное хранилище
- RAG-агент: пользователь задаёт вопрос → генерация ответа со ссылкой на источник
- Без планировщика/фоновых задач

### Критерии готовности MVP

- [ ] Мануал Hyundai Elantra 2004 загружен и проиндексирован
- [ ] На 10+ тестовых вопросов (по обслуживанию, объёмам жидкостей, базовому ремонту) агент даёт релевантные ответы
- [ ] Рабочие инструменты  `Weather_Control` и `Maintenance_Scheduler`
- [ ] Ответ агента содержит указание на источник (раздел/страницу мануала)
- [ ] Если ответа нет в базе — агент честно сообщает об этом, а не выдумывает

## 9. Use Cases

### UC-1: Вопрос по обслуживанию с ответом из мануала
**Актор:** Владелец авто
**Сценарий:** Пользователь пишет боту: «Какое масло заливать в двигатель и какой объём?»
**Действия системы:** Агент делает поиск по разделу «Технические характеристики / Заправочные объёмы» соответствующего мануала → формирует ответ с указанием марки/вязкости масла и объёма, ссылаясь на раздел мануала.
**Результат:** Пользователь получает точный ответ, привязанный к его модели авто.

### UC-2: Подбор запчасти по таблице допусков
**Актор:** Владелец авто
**Сценарий:** «Какой размер дисков можно поставить вместо штатных 17 дюймов?»
**Действия системы:** Агент ищет в таблицах допусков (загруженных как структурированные чанки) допустимые альтернативные размеры и вылет → отвечает списком допустимых вариантов.
**Результат:** Пользователь получает корректные технически допустимые варианты, без риска ошибки посадки.

### UC-3: Напоминание о ТО по пробегу
**Актор:** Система (фоновая задача)
**Сценарий:** Пробег пользователя приближается к порогу очередного ТО согласно регламенту из мануала.
**Действия системы:** `Maintenance_Scheduler` сравнивает текущий пробег/дату с интервалом ТО → создаёт напоминание → Telegram-бот отправляет уведомление: «До ТО осталось 500 км, рекомендуемые работы: замена масла, фильтров».
**Результат:** Пользователь заранее предупреждён о необходимости записаться на ТО.

### UC-4: Рекомендация по смене шин из-за похолодания
**Актор:** Система (фоновая задача)
**Сценарий:** Среднесуточная температура за несколько дней опустилась ниже настроенного порога (например, +5°C).
**Действия системы:** `Weather_Control` фиксирует превышение порога → формирует рекомендацию → отправляет уведомление через Telegram: «Похолодало ниже +5°C — рекомендуется переобуться в зимнюю резину».
**Результат:** Пользователь своевременно меняет шины по сезону.
